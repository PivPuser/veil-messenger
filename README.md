# anon-messenger (рабочее название)

Анонимный мессенджер с политикой **no-logs**, работающий по принципу **ключей**:
пользователь создаёт короткий ключ-приглашение и передаёт его собеседнику; тот
вводит ключ в приложении — и между ними поднимается сквозной зашифрованный канал.
Никаких телефонов, e-mail и логинов. Личность = криптографические ключи,
сгенерированные на устройстве.

> ⚠️ Проект в ранней разработке. Криптографический код **ещё не прошёл аудит** и
> не должен использоваться для защиты реальных секретов, пока не будет проверен
> и отревьюен независимо.

## Принцип «ключей»

- **Чел1** генерирует у себя пару ключей и `signed pre-key`, упаковывает публичные
  части в строку вида `amk1:...` (готова под QR) и отдаёт её **Чел2**.
- **Чел2** вставляет ключ. Приложение проверяет подпись, выполняет обмен **X3DH** и
  сразу поднимает защищённый канал.
- Дальше переписка защищена **Double Ratchet** — у каждого сообщения свой ключ,
  утечка одного не раскрывает остальные (forward secrecy).

## Архитектура (план)

| Слой | Технология | Статус |
|------|-----------|--------|
| Крипто-ядро | Dart, `package:cryptography` (X25519, Ed25519, HKDF, ChaCha20-Poly1305) | ✅ тесты 11/11 |
| X3DH + Double Ratchet | собственный протокольный слой поверх примитивов | ✅ тесты 11/11 |
| Транспорт | «тупой» релей (store-and-forward зашифрованных конвертов) | ✅ тесты + CLI-демо |
| Доступ через Tor | SOCKS5-клиент к Tor + onion-сервис релея | 🟡 клиент готов (10/10 тестов), нужен живой Tor/onion |
| Клиент | Flutter (iOS + Android) | ⏳ дальше |

### Два слоя защиты — и где настоящая сложность

1. **Содержимое** сообщений закрывает E2E-шифрование (этот репозиторий). Это
   решаемая, хорошо изученная задача.
2. **Метаданные** (кто с кем, когда, откуда) — настоящая сложность анонимности.
   E2E их **не** прячет. Поэтому:
   - **no-logs — это архитектура, а не обещание:** релей физически видит только
     зашифрованные конверты со сменными адресами, а не переписку.
   - **IP скрывается через Tor** — иначе релей и провайдер видят реальный адрес.
   - **никакой регистрации** — ни телефона, ни почты.
   - **push-уведомления (FCM/APNs) — известная утечка метаданных**, решается
     отдельно (или отключается).

## Модель угроз (черновик)

**Защищаемся от:**
- пассивного и активного перехвата трафика (в т.ч. оператора релея);
- изъятия/взлома сервера-релея (он не хранит открытых данных и связей);
- компрометации одного сообщения (forward secrecy / post-compromise security).

**Пока НЕ защищаемся от (честно):**
- глобального наблюдателя, коррелирующего тайминги трафика;
- компрометации самого устройства пользователя (malware, разблокированный телефон);
- отсутствия проверки личности собеседника вне приложения — ключ подтверждает
  владение ключом, но не то, что «это точно тот человек» (нужна сверка
  safety-number по отдельному каналу).

## Структура репозитория

```
packages/
  crypto_core/        # E2E-ядро (чистый Dart, тестируется без Flutter)
    lib/src/
      primitives.dart      # обёртки над проверенной крипто-библиотекой
      identity.dart        # долговременные ключи, pre-keys
      invite.dart          # кодирование «ключа» amk1:...
      x3dh.dart            # согласование общего секрета
      double_ratchet.dart  # forward secrecy для сообщений
      session.dart         # высокоуровневый E2E-сеанс (+ сериализация)
      secret_vault.dart    # шифрованное хранилище ключей (AEAD + PBKDF2)
      safety_number.dart   # сверка личности (anti-MITM)
      app_lock.dart        # код-пароль + паник-вайп данных
      codec.dart           # бинарная сериализация
  relay/              # транспорт store-and-forward
    lib/src/
      relay_server.dart      # «тупой» сервер (in-memory, без логов)
      relay_transport.dart   # общий интерфейс клиента
      relay_client.dart      # прямой HTTP-клиент (локально/тесты)
      tor_relay_client.dart  # клиент через Tor SOCKS-прокси
      socks5.dart            # минимальный SOCKS5 (CONNECT по .onion)
      byte_stream_reader.dart
      mailbox.dart           # одноразовые mailbox-id из секрета сессии
    bin/relay_server.dart    # запуск сервера
    bin/tor_smoke.dart       # проверка против живого Tor
  demo/               # CLI-демо полного цикла
    bin/chat_demo.dart
```

## Как запустить

Нужен Dart SDK (входит в состав Flutter).

```bash
# тесты крипто-ядра
cd packages/crypto_core && dart pub get && dart test

# тесты транспорта
cd packages/relay && dart pub get && dart test

# демо полного цикла (Чел1 создаёт ключ, Чел2 вводит, переписка через релей)
cd packages/demo && dart pub get && dart run bin/chat_demo.dart

# отдельный релей-сервер
cd packages/relay && dart run bin/relay_server.dart
```

## Доступ через Tor

Клиент ходит в релей через Tor по SOCKS5 (по умолчанию `127.0.0.1:9050`), а сам
релей публикуется как **onion-сервис** — так ни клиент, ни сервер не раскрывают IP,
а `.onion`-адрес самоаутентифицирует релей.

1. Запусти релей локально (см. выше), слушает, например, `127.0.0.1:8080`.
2. В `torrc` добавь onion-сервис, указывающий на релей:
   ```
   HiddenServiceDir /var/lib/tor/anon_relay/
   HiddenServicePort 80 127.0.0.1:8080
   ```
   Перезапусти Tor, возьми адрес из `hostname` в этой папке.
3. Проверь round-trip через живой Tor:
   ```bash
   cd packages/relay
   ONION=<адрес>.onion dart run bin/tor_smoke.dart
   ```

В коде переключение тривиально — оба клиента реализуют `RelayTransport`:
```dart
final RelayTransport net = TorRelayClient(onionHost: '<адрес>.onion');
// вместо RelayClient(uri) — остальной код (демо/Peer) не меняется
```
